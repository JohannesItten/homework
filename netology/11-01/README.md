### Задание 1. СУБД

### Кейс
Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать 
правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для
каждой предметной области. 

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему? 
 
1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков.
СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы? 

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к 
маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? 
СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу 
и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это 
реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов 
по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать
со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД 
логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

*Приведите ответ в свободной форме.*

### Ответ 1. СУБД
#### 1.1 Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков

Реляционная СУБД, т.к.:
- строгая схема данных
- ACID транзакции = целостность данных
- SQL для аналитических запросов

#### 1.2. Под каждый девелоперский проект создаётся отдельный лендинг...

CRM - реляционная СУБД, по причинам озвученным в 1.1
Лендинги - NoSQL, т.к. у каждого свои особенности, например, поля формы обратной связи

#### 1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

Можно закрыть, например, PostgreSQL. Всевозможные нюансы лендингов хранить как JSONB 

#### 1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам...

Документоориентированная СУБД, т.к.:
- Нормы, регламенты, статьи - это документы с вложенной структурой
- Полнотекстовый поиск по содержимому из коробки
- Иерархия компании удобно ложится в вложенные объекты

#### 1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше...

Да, например, PostgreSQL. Проще поддержка, единое пространство данных, проще бэкапы и т.д.

#### 1.4 Департамент логистики нуждается в решении задач по быстрому формированию маршрутов 

Графовая СУБД, т.к. маршрут = граф

#### 1.4.* Можно ли к этой СУБД подключить отдел закупок  

Можно, но не нужно. Т.к. закупки/финансы - транзакционные данные

#### 1.5 Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

Да, например, PostgreSQL. Можно все и в CSV хранить, вопрос задач и разумности решений.

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

*Приведите ответ в свободной форме.*

### Ответ 2
#### 2.1 Пользователь пополняет баланс счёта телефона, распишите пошагово...

#### Шаг 1, Инициация транзакции
Пользователь вводит номер телефона и сумму, нажимает «Оплатить». Система создаёт транзакцию
#### Шаг 2, Валидация данных
Проверка корректности данных до обращения к банку
#### Шаг 3, Авторизация платежа в банке
Система отправляет запрос в банк/платёжный шлюз, валидация внутри банковской системы
#### Шаг 4, Запрос к оператору связи
Платёжная система отправляет запрос оператору через API. Передаем номер, сумму пополнения и тд
#### Шаг 5, Подтверждение и проведение транзакции
Банк списывает средства, оператор производит зачисление
#### Шаг 6, Уведомление пользователя
Уведомление пользователя и всех сторон транзакции

#### 2.1.* ...если пополнение счёта телефона происходило бы через автоплатёж?

Меняем Шаг 1 на проверку настроек автоплатежа. Если все ок, то идем по шагам выше.

### Ответ 3

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. 

На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?

*Приведите ответ в свободной форме.*

### Ответ 4
Из условий задачи и CAP делаем вывод, что доступность и устойчивость важнее строгой консистентности.
Колоночная СУБД + Apache Spark (MapReduce через RAM): 
- для аналитических вычислений над миллиардами строк колоночное хранение даёт кратный выигрыш, потому что читаются только нужные колонки, а не вся строка целиком
- Apache Spark как раз и создавался под подобные задачи, хранит промежуточные данные в RAM, а не на диске
